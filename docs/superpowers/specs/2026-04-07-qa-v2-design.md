# QA Swarm v2 — Design Spec Revisado
**Data:** 2026-04-07  
**Projeto:** SuperAgentes B2C V2  
**Status:** Aprovado para execução amanhã  
**Motivo da revisão:** QA v1 (2026-04-06) foi bloqueado pelo limite de 25 interações compartilhado entre sessões paralelas. Toda a bateria ficou inconclusiva. Esta versão resolve isso com 3 fluxos sequenciais independentes + reset automático de contador.

---

## O problema central do QA v1

O QA anterior rodou 5 agentes Playwright em paralelo na mesma conta. O contador `b2c_uso_diario` é compartilhado por família — 5 sessões simultâneas consomem o mesmo pool de 25. Com A1 (PAI/Layla ativa), A2 (PAI/Layla ativa), A3 (Layla FILHO), A4 (Maria Paz) e A5 (Layla) rodando juntos, o limite de Layla foi a 26 em ~20 minutos. Resultado: testes bloqueados antes de completar qualquer perfil.

A solução não é aumentar o limite permanentemente — é **resetar o contador antes de cada bateria de perfil** via Supabase durante o QA. Nenhuma mudança de código necessária.

---

## Arquitetura v2 — 3 Fluxos Sequenciais

```
[FLUXO 1] Layla (7º ano)
  └─ Reset contador → bateria completa → relatório Layla
  └─ ~35 interações esperadas

[FLUXO 2] Maria Paz (3º ano)
  └─ Reset contador → bateria completa → relatório Maria Paz
  └─ ~25 interações esperadas

[FLUXO 3] PAI (Leon)
  └─ Reset contador (Layla ativa, depois Maria Paz ativa) → bateria completa → relatório PAI
  └─ ~30 interações esperadas

[PÓS-EXECUÇÃO] Agente Psicopedagógico
  └─ Analisa as 3 transcrições → relatório consolidado
```

---

## Protocolo de Reset de Contador

Antes de iniciar cada fluxo, executar via Supabase MCP:

```sql
-- Para Layla (aluno_id = '0fb1c38f-7d34-45b1-8ff2-3e5c4ccff71e'):
UPDATE b2c_uso_diario SET interacoes = 0 
WHERE aluno_id = '0fb1c38f-7d34-45b1-8ff2-3e5c4ccff71e' 
AND data = CURRENT_DATE;

-- Para Maria Paz (aluno_id = '6fa15ba4-38e8-4628-9b51-e0a3076d631a'):
UPDATE b2c_uso_diario SET interacoes = 0 
WHERE aluno_id = '6fa15ba4-38e8-4628-9b51-e0a3076d631a' 
AND data = CURRENT_DATE;
```

**Reset periódico:** Se um fluxo tiver mais de 20 interações, resetar na metade para garantir margem. Verificar com `SELECT interacoes FROM b2c_uso_diario WHERE aluno_id = '...' AND data = CURRENT_DATE`.

**Pré-condição:** Garantir que o registro existe para hoje antes de atualizar. Se não existir, o primeiro reset pode ser omitido (começará do zero naturalmente).

**Reset de contador: SIM (antes de iniciar)**

- Turno 2: follow-up → TEMPUS direto (sem PSICO), tempo <3s?

### Bloco 1.2 — Rota por botão → CALCULUS
- Limpar/reiniciar chat (ou abrir nova sessão para Layla)

### Bloco 1.3 — Construtivismo (TESTE CRÍTICO)
- Com CALCULUS ativo: input "quanto é 2/3 + 1/4?"

### Bloco 1.4 — Troca de tema
- Após 3 turnos em Matemática: "agora quero falar de português"

- Turno 2-3 em VERBETTA

### Bloco 1.5 — Pedido explícito de herói
- Input: "quero falar com o GAIA sobre o Brasil"
- GAIA ativa? Header mostra GAIA?

### Bloco 1.6 — Imagem
- Tentar upload via botão `+`

## Testes de UI (Independentes dos fluxos — rodar ao final)

Estes não dependem de interações com IA, não consomem contador:

- Banner de limite: verificar se já havia banner visível → dismiss com ✕
- Logout: Sair → sa_token = null → redirect /login
- Shift+Enter: não envia, cria nova linha
- Enter: envia
- Send + Anexar disabled durante streaming
- Send habilitado ao digitar
- Token persistido após reload (sa_token presente)
- Console errors: zero em todas as sessões

## Avaliação do Limite de 25 Interações/Dia

Esta seção deve ser concluída após o Fluxo 1 completo de Layla.

### Dados a coletar durante o Fluxo 1
Contar quantas interações uma sessão de estudo real consome:
- Turno 1 de uma matéria nova: 1 interação
- Follow-ups por tópico: ~2-3
- Troca de matéria: 1 nova interação (cascade) + 2-3 follow-ups
- Uma sessão completa cobrindo 2 matérias: ~8-14 interações

### Avaliação qualitativa (preencher após o fluxo)
- Uma sessão de estudo típica (1 matéria, 4 turnos) = X interações
- Uma sessão robusta (2 matérias + revisão) = Y interações
- Limite de 25/dia é suficiente para uma sessão diária? SIM / NÃO
- Se pai usa 10 turnos (orientação) + filho usa 15 (estudo) = 25 — justo na borda

### Recomendação de ajuste (preencher após análise)
- [ ] 25 é adequado: nenhuma mudança
- [ ] 30 seria mais justo: ajustar em servidor/src/config
- [ ] O problema real é PAI e FILHO compartilharem o mesmo contador → fix no código é a solução correta

## Melhorias de Processo propostas por OSCAR

Além dos ajustes de Leon, estas são minhas propostas baseadas na execução do QA v1:

**1. Reset automático a cada 20 interações** — não esperar o limite. No início de cada bloco de mais de 5 turnos, verificar o contador e resetar se acima de 18.

**2. Timeout de streaming ajustado** — respostas levaram 8-10s mesmo em turnos de continuidade (esperado <3s). Investigar se o PSICO está rodando de novo em turnos de continuidade (seria um bug de performance ou design). Medir separadamente: tempo até primeiro token vs. tempo total de stream.

**3. Verificação de JSON leak em todos os turnos** — hoje verifiquei manualmente. No v2, incluir `playwright-cli eval "document.body.innerText"` e procurar por `"agente":` ou `{\"tipo\":` em cada resposta.

**4. Construtivismo para TODOS os heróis, não só CALCULUS** — no v1 só testei CALCULUS. No v2, incluir teste construtivista para cada herói na bateria do perfil correspondente. Exemplo: TEMPUS não deve resolver "quem ganhou a 2ª Guerra?" direto — deve perguntar o que Layla já sabe.

**5. Comparação de tempo botão vs. semântico registrada em log** — hoje medi manualmente. No v2, capturar em JSON e comparar na conclusão do fluxo.

**6. Teste de "herói substituto"** — quando VERBETTA está ativa e aluno pergunta matemática, o herói deve redirecionar para CALCULUS ou o próprio aluno precisa pedir? Isso define se o PSICO é apenas de estado neutro ou também atua em transições.

**7. Transcrições em JSON por turno (obrigatório)** — no v1 capturei pelo snapshot. No v2, salvar JSON por turno com: input, output_literal, herói_header, tempo_ms, json_leak, console_errors, construtivista_sim_nao. Esse JSON é o input do agente psicopedagógico.

## Critério de Aprovação (mantido da v1)

- Zero bugs críticos abertos
- Zero bugs maiores em fluxos principais (auth, cascata, MODO PAI ESTADO A)
- Score psicopedagógico ≥ 8/10 em todos os heróis testados em MODO FILHO
- Score psicopedagógico ≥ 8/10 nos heróis em MODO PAI
- Construtivismo aprovado: CALCULUS não resolve "quanto é X?" diretamente
- Todos os itens do scorecard com ✅ PASS
